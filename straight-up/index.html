Tested Software Versions

  • OSX 10.9.5
  • Safari 7.1.6
  • Chrome 42.0.2311.152

VoiceOver, in Safari

Voiceover is surprisingly chatty on first interacting with the page. "Interact with Straight Up HTML | A11y Experiments HTML content heading level 1 A11Y Experiments: Straight Up, Valid HTML5. Interact with heading level 1 A11Y Experiments: Straight Up, Valid HTML5. A11Y Experiments: Straight Up, Valid HTML5."

Though "header" and "nav" tags are used, Safari does not announce that we are entering a banner landmark or a navigation landmark.

In the nav, Voiceover focuses on each bullet and speaks "pillet" before focusing on and speaking the text of the bullet item. Contrast this with its behavior reading the sample unordered list, and the unordered list of software versions above. When a link begins the list item text, the bullet receive focus, and is spoken as "pillet." If normal text begins the list item, the bullet is grouped with the content, and does NOT receive separate focus. Instead, VoiceOver says "bullet OSX10.9.5", etc.

No obvious difference in the way strong text, emphasized text, or code is spoken. It's possible this can be configured in VoiceOver settings, but is likely an option reserved for power-users if so.

The link to Google works as expected, but the link to the ordered list does not. Keyboard focus remains on the link, and VoiceOver is silent. After advancing again, VoiceOver reads the second ordered list item, implying that partially successful refocusing occured.... but not particularly good refocusing.

Column headers are correctly read before cell table contents

Form groups are announced as groups, allowing you to interact with the elements or not, as desired.

Labels receive focus and are read separately, and then are read again when focusing on the input element

VoiceOver speaks "blockquote level 1" after speaking the block quote.

The code is simply read through.

When entering the footer, VoiceOver announces that you are entering the footer landmark.

VoiceOver, in Firefox

On interacting with the page, VoiceOver does announce that we are in the "banner" landmark. It's doesn't repeat itself, like it does in Safari.

On entering the nav element, VoiceOver says we are entering the navigation landmark. Unlike in Safari, bullets do not receive separate focus. Instead, each item is read, "link One bullet One."

The "main" landmark is announced, when we enter main. The text is read with no discernable difference between regular and strong, etc., and the internal link doesn't work at all. The screen scrolls a bit, but keyboard and VoiceOver remains on the link. So, when you advance, its as though you had never clicked at all, but with a confusing screen scroll.

For the non-link list item, the text is read alone, as we'd expect. The linked list item is read like the nav links "This is a.... bullet. This is a...". But, VoiceOver speaks "double quote" at the end of the line, for some reason, when reading the text-list items. (Line-ending encoding issue? No idea....).

Table rows are read as units, each column one after another, with no indication of column breaks. Each row is called a "list box." You can interact with each row (each list box) if you want to hear each cell individually. Each cell is then called a group (e.g. "Kramer group"). The column headers are not read before the cell content. The VO+C results in an error, not in the column header being read. Weird.

Each label in the form receives focus separately. Then, when entering the input element, each label is read , and then the legend is read twice. VO+space does not toggle the radio buttons, though VoiceOver says "press". Space pressed alone, without the VO keys, does toggle the radio button, but VoiceOver remains silent. VO+space works normally with checkboxes.

The select menu was impossible to use. VO+space did open the menu, but then it was impossible to move the focus off "hot." Nor did anything else I tried work. Very erratic.

Blockquote not announced as such.

VoiceOver, in Chrome

On interacting with the page, VoiceOver does announce that we are in the "banner" landmark. It's doesn't repeat itself, like it does in Safari.

On entering the nav element, VoiceOver says we are entering the navigation landmark. However, like Safari, and unlike Firefox, Voiceover focuses on each bullet and speaks "pillet" before focusing on and speaking the text of the bullet item. Further, it says "clickable" after saying "pillet"! If you press the link (VO+space) it says, "press list", and thankfully, doesn't take you anywhere.

The "main" landmark is announced, when we enter main. The text is read with no discernable difference between regular and strong, etc., and the internal link doesn't work at all. The screen scrolls a bit, but keyboard and VoiceOver remains on the link. So, when you advance, its as though you had never clicked at all, but with a confusing screen scroll. Oddly, it says "double quote" at the end of the paragraph. No idea what's up there, again.

List items as in Safari and in the nav element.

The table is identified as a table, and not read bizarrely as in Firefox, but cell contents are not read properly with their column headers. Oddly, the first column, ID, is. The first column is also the only column in which column and row numbers are being read (despite VoiceOver config). VO+C does not announce the column header, or give an error.

Forms are great. When entering a fieldset, you are read the legend. Then, you are brought to the first input element, and read the label, then the legend, then the input state ("tea, beverage choice, unselected radio button.). Ideal.

Blockquote not announced as such

ChromeVox, in Chrome

Much less verbose. Only heading is read, and then it says "heading one". Doesn't mention a banner role, doesn't mention the page title.

Says, "navigation with eight items. One, link item. Two, link list item." Etc.

Announces entering a section. Doesn't say anything about "main".

Reads paragraph as a whole or one screen-line at time, including links, without stopping. Like all other cases, no discernable difference in strong or emphasized text or code. After the line is read, e.g. while it's highlighted, you can press tab to get to the links. You DO end up at the ordered list, when you click the link! It's read properly.

Lists are read really well. "Unordered list with two items" and then the list item text. No "bullet" or "pillet" or weird focusing, even when the list item is a big link. Instead, it just reads the link information like you'd expect, e.g., by saying "internal link" after reading the text.

Like Firefox, table rows are read as groups (though it doesn't say anything like "list box"). You can enter table mode for more normal nav. In table mode, headers are not read along with cell contents.

Forms are navigated slightly differently (ChromeVox+right/ left for fieldsets, ChromeVox+up/down for options within), but are generally read in an intuitive way, like with Chrome and VoiceOver, once you figure that out. The combobox, however, I couldn't figure out. I could activate the menu, but couldn't seem to get it to read me options.

"Section, Discussion" is announced. And again, the lists are read well, without odd focusing or "pillet" or "bullet" being spoken.
